# DASM Platform — Control Log

مرجع تشغيلي دائم للمشروع. يُحدّث مع كل دفعة عمل رئيسية.

هذا الملف هو المرجع التشغيلي الأعلى للمشروع، ويُعتمد عند تعارض الذاكرة أو التقارير المختصرة أو الحالة المتخيلة مع الواقع الحالي في المستودع والإنتاج.

---

## 1. الغرض من الملف

- أن يكون **مرجع العمل الأعلى** (source of truth) للنشاط التشغيلي للمنصة
- توثيق القواعد والسياسات المعتمدة
- تتبع baseline الحالة الحالية والسجل الزمني للدفعات
- تحديد المهام النشطة والمؤجلة والأولويات

---

## 2. القواعد التشغيلية الثابتة

| القاعدة | الوصف |
|---------|-------|
| **Branch → PR → merge** | لا يتم الدفع مباشرة إلى `master`. كل تعديل عبر فرع ثم PR ثم مراجعة ودمج. |
| **Minimal Safe Production Fix** | التصحيحات الإنتاجية تكون بالحد الأدنى: إصلاح ما هو mock أو broken فقط، دون توسع غير ضروري. |
| **فصل نطاقات العمل** | إصلاحات Dashboard منفصلة عن إصلاحات البنية (auth/rewrite/env). كل منهما في PR مستقل. |
| **لا أرقام وهمية** | الإحصائيات والعروض تستخدم بيانات حقيقية فقط. إن تعذر — إخفاء العنصر أو خصم unavailable. |
| **لا روابط مكسورة** | أي quick action أو رابط يؤدي إلى صفحة غير مكتملة يُخفى أو يُزال حتى يُنفّذ. |
| **لا تكرار غير ضروري** | إزالة التكرار في وصول المستخدم (مثل ازدواج زر الإعدادات/الملف الشخصي). |

---

## 2.5 سياسة نسبة العمل في الـ PRs والـ Commits (PR / Commit Attribution Policy)

هذا القسم جزء من الانضباط التشغيلي للمستودع، ويهدف إلى إبقاء السجل نظيفًا ومتسقًا.

| القاعدة | الوصف |
|---------|-------|
| **لا إشارة إلى أدوات في السجل** | لا نضع أي إشارة إلى Cursor أو أي أداة مساعدة داخل commit messages أو PR descriptions أو merge descriptions. |
| **حذف النصوص المولدة تلقائيًا** | يجب حذف أي نص مضاف تلقائيًا مثل `Made with Cursor` أو ما شابهه **قبل** تنفيذ الدمج. |
| **صيغة نسبة العمل عند الحاجة** | عند الحاجة الفعلية إلى attribution، الصيغة المعتمدة: *Planned, reviewed, and approved by mazroni9*. |
| **الأفضلية للمحايدية** | إذا لم تكن هناك حاجة فعلية إلى attribution، يُفضّل ترك العنوان والوصف مهنيين ومحايدين دون أي نسبة إلى أداة. |

---

## 2.6 سياسة تحسينات UX (برنامج الفرع الطويل العمر)

| القاعدة | الوصف |
|---------|-------|
| **فرع program/ux-improvements** | فرع طويل العمر مظلة لتحسينات الواجهات وتجربة الاستخدام على مستوى المنصة. لا يُحذف بعد أول دمج أو دفعة واحدة؛ يبقى قائمًا حتى انتهاء حزمة كافية من تحسينات UX واستقرارها. |
| **ربط تحسينات UX** | أي تحسينات UX مستقبلية في الواجهة العامة أو جميع الداشبوردات التابعة للمنصة يجب أن ترتبط بهذا البرنامج. |
| **انضباط الدفعات** | لا يعني ذلك جمع تغييرات غير منضبطة أو دفعات ضخمة بلا مراجعة؛ كل دفعة قابلة للمراجعة والاختبار. |

## 3. Baseline الحالة الحالية

| المكون | الحالة |
|--------|--------|
| **Frontend** | Next.js على Vercel (Production + Preview) |
| **Backend** | Laravel على Render (`dasm-platform-backend`) |
| **Dashboard stats** | حقيقية عبر `GET /api/dashboard/stats` |
| **My Sales** | حقيقية عبر `GET /api/settlements?seller=me` |

| **صلاحيات council.\*** | مُعرّفة في `RolesAndPermissionsSeeder`؛ `DatabaseSeeder` يستدعي هذا الـ seeder. الصفحة `/admin/market-council/permissions` تعمل بعد زرع الصلاحيات. حالة empty state السابقة كانت ناتجة عن غياب seed في البيئة وليست نقصًا في التعريف بالكود. |
| **برنامج UX** | اعتماد `program/ux-improvements` كفرع طويل العمر لبرنامج تحسين UX. |

## 6. المهام النشطة الحالية

- مراقبة استقرار Production بعد تحديث ربط API على api.dasm.com.sa
- التحقق من login والجوال بعد آخر تعديلات Vercel env
- مراجعة نهائية لتجربة الجوال بعد PR #196
- استخدام صفحة «صلاحيات مجلس السوق» الجديدة لتعيين الصلاحيات بدل الفحص اليدوي
- مراقبة استقرار التنقل في `/admin/market-council/permissions` خصوصًا تحت الشبكات الضعيفة
- مراجعة ودمج PR #201 إن تم اعتماد اتجاه «الحسابات المميزة فقط» افتراضيًا؛ الصفحة أداة للحسابات المميزة/الاستثناءات وليست بديلًا كاملًا لصفحة إدارة المستخدمين
- **بدء برنامج إصلاح UX للمنصة** عبر `program/ux-improvements` ليشمل الصفحة الرئيسية، الواجهات العامة، وجميع الداشبوردات التابعة للمنصة
- **أولوية قصوى:** إصلاح تصفح المنصة من متصفح الجوال، بعد ظهور خطأ خطير يمنع رؤية المنصة عبر المتصفح من الجوال
- **واجهة اقتراحات وآراء المستخدمين:** مكان مستقل في واجهة المنصة لاستقبال اقتراحات المستخدمين وآرائهم وتقييماتهم للمنصة، مع إرسالها إلى لوحة الكنترول روم. يُحلل يدويًا/شخصيًا وكذلك عبر الذكاء الاصطناعي، مع فرز المقترحات وأخذ الصالح منها. يُفضّل أن يكون مستقلاً عن مجلس السوق، ووُضِع في جزء غير حرج من الصفحة الرئيسية (مثل: «آراء زوار ورواد DASM-e»)
- **برمجة الفحص عبر الورش محليًا** مع تهيئته لاحقًا ليصبح تطبيقًا على iPhone وAndroid؛ يُبنى داخليًا ثم يُربط لاحقًا عبر API
- **تتبع واسترجاع تاريخ «مجلد الذكاء الصناعي»** الذي كان موجودًا في المشروع ثم حُذف؛ تحديد عبر أي PR دخل أولًا وعبر أي PR خرج أو حُذف لاحقًا؛ اعتماد ذلك كمسار فحص لاحق مهم
- **متابعة ملف إصلاح بعض المشاكل التقنية الحالية**
- **إكمال مشروع الاختبارات الحالي وإغلاقه** وفق ملف الاعتماد/التسليم المطلوب في مشروع منصة مستقل
