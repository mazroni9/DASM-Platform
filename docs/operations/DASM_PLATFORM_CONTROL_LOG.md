# DASM Platform — Control Log

مرجع تشغيلي دائم للمشروع. يُحدّث مع كل دفعة عمل رئيسية.

هذا الملف هو المرجع التشغيلي الأعلى للمشروع، ويُعتمد عند تعارض الذاكرة أو التقارير المختصرة أو الحالة المتخيلة مع الواقع الحالي في المستودع والإنتاج.

---

## 1. الغرض من الملف

- أن يكون **مرجع العمل الأعلى** (source of truth) للنشاط التشغيلي للمنصة
- توثيق القواعد والسياسات المعتمدة
- تتبع baseline الحالة الحالية والسجل الزمني للدفعات
- تحديد المهام النشطة والمؤجلة والأولويات

---

## 2. القواعد التشغيلية الثابتة

| القاعدة | الوصف |
|---------|-------|
| **Branch → PR → merge** | لا يتم الدفع مباشرة إلى `master`. كل تعديل عبر فرع ثم PR ثم مراجعة ودمج. |
| **Minimal Safe Production Fix** | التصحيحات الإنتاجية تكون بالحد الأدنى: إصلاح ما هو mock أو broken فقط، دون توسع غير ضروري. |
| **فصل نطاقات العمل** | إصلاحات Dashboard منفصلة عن إصلاحات البنية (auth/rewrite/env). كل منهما في PR مستقل. |
| **لا أرقام وهمية** | الإحصائيات والعروض تستخدم بيانات حقيقية فقط. إن تعذر — إخفاء العنصر أو خصم unavailable. |
| **لا روابط مكسورة** | أي quick action أو رابط يؤدي إلى صفحة غير مكتملة يُخفى أو يُزال حتى يُنفّذ. |
| **لا تكرار غير ضروري** | إزالة التكرار في وصول المستخدم (مثل ازدواج زر الإعدادات/الملف الشخصي). |
| **مراجعة Control Log إلزامية قبل أي عمل** | لا يُبدأ أي عمل جديد في المشروع (بما في ذلك عبر أدوات المساعدة) قبل مراجعة `docs/operations/DASM_PLATFORM_CONTROL_LOG.md`. الملف هو **المرجع التشغيلي الأعلى**. عند تعارض أي ذاكرة أو توجيه مختصر أو افتراض سابق مع اللوق، تكون **الأولوية دائمًا للّوق**. أي مهمة جديدة تُسجَّل أو تُراجع مقابل اللوق قبل التنفيذ. |

## 2.5 سياسة نسبة العمل في الـ PRs والـ Commits (PR / Commit Attribution Policy)

هذا القسم جزء من الانضباط التشغيلي للمستودع، ويهدف إلى إبقاء السجل نظيفًا ومتسقًا.

| القاعدة | الوصف |
|---------|-------|
| **لا إشارة إلى أدوات في السجل** | لا نضع أي إشارة إلى Cursor أو أي أداة مساعدة داخل commit messages أو PR descriptions أو merge descriptions. |
| **حذف النصوص المولدة تلقائيًا** | يجب حذف أي نص مضاف تلقائيًا مثل `Made with Cursor` أو ما شابهه **قبل** تنفيذ الدمج. |
| **صيغة نسبة العمل عند الحاجة** | عند الحاجة الفعلية إلى attribution، الصيغة المعتمدة: *Planned, reviewed, and approved by mazroni9*. |
| **الأفضلية للمحايدية** | إذا لم تكن هناك حاجة فعلية إلى attribution، يُفضّل ترك العنوان والوصف مهنيين ومحايدين دون أي نسبة إلى أداة. |

## 2.6 سياسة تحسينات UX/UI البصرية فقط (الفرع الطويل العمر الرسمي)

| القاعدة | الوصف |
|---------|-------|
| **الفرع الرسمي** | `program/ux-improvements-2` هو الفرع طويل العمر **الوحيد** المعتمد لتحسينات **الواجهات البصرية فقط**: التنسيقات، البطاقات، المسافات، ترتيب العناصر، polish بصري للجوال، واجهة الصفحة الرئيسية، الداشبورد، مجلس السوق، وأي تحسين **مظهري** لا يغيّر عقود API أو قاعدة البيانات. |
| **ممنوع على هذا الفرع** | أي من: تغييرات **backend**، **قاعدة بيانات**، **seeders**، **migrations**، **API**، **auth**، **runtime / providers**، **إصلاحات crash جوال مرتبطة بـ Firebase أو منطق تشغيل**، أو أي إصلاح **تقني** غير بصري. تلك الأنواع تذهب إلى **فرع قصير العمر مستقل** ثم **PR مستقل**. |
| **انضباط الدفعات** | كل دفعة بصرية قابلة للمراجعة والاختبار؛ لا دمج تغييرات تقنية ضمن برنامج UX البصري. |

### 2.6.1 تصحيح تنظيمي (الفرع السابق `program/ux-improvements`)

- استُخدم الفرع `program/ux-improvements` بشكل **مختلط** ودخلت فيه إصلاحات **تقنية** (مثل Firebase / runtime) لا تنتمي إلى **UX البصري الصرف**.
- تم اعتماد `program/ux-improvements-2` **لتصحيح هذا الخلط** وفرض نطاق بصري صريح.
- **من الآن فصاعدًا** يُمنع خلط إصلاحات backend / database / runtime مع برنامج تحسينات الواجهة البصرية على الفرع الطويل العمر.

### 2.6.2 حالة الفرع القديم `program/ux-improvements` والحذف

- **PR #204** مدمج في `master` (يشمل إصلاح تصفح الجوال المتعلق بـ Firebase Messaging وتحسينات بصرية للصفحة الرئيسية ضمن دمج واحد).
- دُمج جزء من التعديلات المرتبطة بمجلس السوق لاحقًا في `master` عبر **PR #207** (تخفيف بداية الصفحة) و**PR #208** (تصنيف في الـ seeder). يُراجع عند الحاجة ما إذا بقي على الفرع `program/ux-improvements` في الأصل commits غير مدمجة أو مكررة مع ما دُمج، ثم يُقرّر دمج المتبقي أو حذف الفرع بعد التحقق.
- **لا يُحذف** الفرع `program/ux-improvements` من GitHub حتى التحقق من عدم بقاء عمل معتمد عليه فقط على ذلك الفرع (انظر أعلى).
- بعد التحقق وعدم الحاجة للفرع، يُحذف من الأصل لإغلاق المسار المختلط.

## 3. Baseline الحالة الحالية

| المكون | الحالة |
|--------|--------|
| **Frontend** | Next.js على Vercel (Production + Preview) |
| **Backend** | Laravel على Render (`dasm-platform-backend`) |
| **Dashboard stats** | حقيقية عبر `GET /api/dashboard/stats` |
| **My Sales** | حقيقية عبر `GET /api/settlements?seller=me` |
| **API rewrites** | Production: `API_BACKEND_URL` = `https://api.dasm.com.sa`؛ قابلة للتكوين |
| **/ping** | Route Handler محلي يعيد 200 |
| **التسجيل (Frontend)** | يمر عبر عميل API موحّد (shared axios) وطلبات same-origin إلى `/api/register` — لا مسار تسجيل منفصل بعميل مختلف عن بقية الواجهة. |
| **المستخدم العادي (`user`)** | لا ينتظر تفعيلًا إداريًا بعد التسجيل ضمن السياسة الحالية؛ ما يزال يحتاج توثيق البريد قبل الدخول حسب التدفق المعتمد. |
| **أنواع أخرى (مثل dealer / venue_owner / investor)** | بعد توثيق البريد تُنشأ لها طلبات موافقة تشغيلية (`business_account`) وتنتقل إلى مسار المراجعة وفق النوع والسياسة. |
| **مجموعة الموافقات التشغيلية (Operational approval group)** | مفهوم منفصل عن Spatie teams/roles: جدول `approval_group_members` و`approval_requests` وسجل تدقيق `approval_request_logs`. **كل مسارات `/admin/*` (بما فيها غرفة المعالجة) لموظفي الإدارة فقط:** `super_admin`، `admin`، `moderator`، `programmer`. عضوية المجموعة **لا تقبل** مستخدمًا عاديًا أو تجاريًا في الـ backend؛ طابور الـ API والإشعارات تتطلب نوعًا إداريًا + `is_active` + `can_review_requests`، وإشعارات الطلبات الجديدة تُرسل فقط لمن لديه صلاحية نوع الطلب (`can_approve_business_accounts` / `can_approve_council_requests`) حسب نوع الطلب. إدارة العضوية لـ `super_admin` فقط من `/admin/control-room/approval-group`. طلبات مجلس السوق: `POST /api/council-studio/access-request`. |
| **verify-email وإعادة إرسال التحقق** | تعتمد على عميل API موحّد وطلبات same-origin إلى مسارات `/api` — لا اعتماد على استدعاء مباشر من المتصفح بعنوان backend منفصل من صفحة التحقق. |
| **حذف المستخدمين (لوحة الإدارة)** | محصور على `super_admin` في الواجهة والخادم؛ ممنوع حذف `super_admin` أو حذف الحساب الحالي للمُدير. |
| **دور `programmer`** | مضاف في النظام ومسارات الإدارة؛ مزامنة Spatie تُضبط عبر migrations + provisioner مخصص — **لا** يُشترط تشغيل `php artisan db:seed --class=RolesAndPermissionsSeeder` كخطوة نشر خاصة بهذه الحزمة. |
| **صلاحيات council.\*** | مُعرّفة في `RolesAndPermissionsSeeder`؛ `DatabaseSeeder` يستدعي هذا الـ seeder عند الزرع الكامل. الصفحة `/admin/market-council/permissions` تعمل بعد زرع الصلاحيات. حالة empty state السابقة كانت ناتجة عن غياب seed في البيئة وليست نقصًا في التعريف بالكود. |
| **نموذج الهوية والأدوار** | التصميم الحالي يعتمد أساسًا على `users.type` كهوية تشغيلية واحدة للحساب؛ وجود Spatie لا يحول ذلك إلى multi-role identity نظيف. التشغيل الحالي يفترض فصل الشخصيات المختلفة بحسابات مختلفة إلى حين إعادة تصميم صريحة إن طُلب multi-role حقيقي لاحقًا. |
| **برنامج UX البصري** | الفرع الطويل العمر الرسمي: `program/ux-improvements-2` (بصري فقط؛ انظر القسم 2.6). الفرع `program/ux-improvements` **موقوف عن الاستخدام** بعد التصحيح التنظيمي؛ لا يُحذف من الأصل حتى دمج المتبقيات (انظر 2.6.2). |

### 4.1 PR #220 — حدود النطاق، ما قبل الدمج، والمتابعة

| الموضوع | الوصف |
|---------|--------|
| **نطاق اللوق الحالي** | `bid_sensitive_context_logs` يُنشأ **فقط بعد نجاح** حفظ المزايدة. لا يغطي محاولات مرفوضة (صلاحية، رصيد، تحقق من السعر، إلخ) ولا سلوكًا مشبوهًا قبل إتمام السجل — ذلك **متعمد في هذه الدفعة** كأساس آمن وبسيط. |
